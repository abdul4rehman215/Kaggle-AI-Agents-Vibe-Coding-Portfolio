## What is committed

This folder includes:

- human-readable study notes,
- compact key-concept notes,
- a study guide summary,

The following are intentionally excluded:

- raw NoteGPT output files,
- raw NotebookLM output files,
- the full whitepaper PDF,
- full codelab PDFs,
- copied podcast transcripts,
- copied codelab pages,
- private email screenshots,
- private account screenshots,
- Google Cloud billing screenshots,
- project IDs or account-specific cloud details,
- API keys or credentials,
- `.env` files,
- generated cloud config files,
- live endpoint URLs,
- runtime IDs,
- deployment logs,
- console screenshots,
- and any evidence that was not produced through actual execution.

## About the infographics

The two infographics in `assets/infographics/` are visual study assets created during revision.

They are useful for explaining the Day 5 mental model, but they are not official course diagrams. I am keeping them as personal learning artifacts, similar to visual revision notes.

---

## Why the raw generated notes are not committed

The generated podcast and study-guide notes were useful for revision, but pasting them directly into the repo would make the folder repetitive and less personal.

For this portfolio, I want the notes to show what I understood, not just what a summarizer produced.

The documentation therefore uses the generated notes as input material and converts them into concise, human-readable study notes.

---

## Codelab documentation boundary

The optional cloud codelabs are documented as reviewed architecture walkthroughs.

If I run them later in a prepared cloud environment, the follow-up documentation should add only real artifacts:

- commands that were actually run,
- safe screenshots,
- troubleshooting notes,
- cleanup notes,
- deployment observations,
- and notes about any deviations from the official flow.

No fake completion evidence should be added just to make the folder look full.
