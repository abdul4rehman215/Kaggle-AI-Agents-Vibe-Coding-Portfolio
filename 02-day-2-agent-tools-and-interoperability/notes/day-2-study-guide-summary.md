## ❓ Questions that helped me check understanding

These were the types of questions that made the topic clearer:

### What problem does MCP solve?

MCP reduces the need for custom wrappers between every model and every tool. Instead of building a custom connector for each pair, models and tools can communicate through a shared protocol layer.

### Why is A2A not the same as MCP?

MCP is for tools and data sources. A2A is for communication between agents. A tool usually has a bounded input/output shape. A specialist agent may need a multi-turn conversation.

### Why does A2UI avoid generating arbitrary frontend code?

Because executable code generated by a model can create security problems. A2UI is safer because the agent declares UI intent and the client renders it using trusted components.

### Why do UCP and AP2 need to be separate?

UCP is about commerce interaction, such as browsing menus, checking availability, and building orders. AP2 is about payment authorization and ensuring the transaction follows human-approved rules.

### What is a mandate?

A mandate is a rule boundary for payment. It tells the system what the agent is allowed to spend, where, and under what conditions.

## 🧠 Concepts that clicked after multiple passes

### 1. Protocols prevent integration debt

Before this unit, I understood tools mostly as useful add-ons. Now I understand that tool access without standards can become a maintenance problem.

If every tool needs a different custom wrapper, the developer becomes stuck maintaining fragile adapters instead of building the actual product.

### 2. One giant agent is not a scalable architecture

The monolithic agent idea is tempting because it feels simple: give one agent all tools and instructions. But after reading the paper, I understand why that can fail.

Too many tools and schemas create a large action space. The agent may call the wrong tool, use the wrong parameters, or lose focus.

Specialized agents and smaller toolsets make the system easier to reason about.

### 3. Tools and agents have different interaction shapes

This was probably the most important distinction.

A tool is usually predictable. A specialist agent is more like a collaborator. It may ask questions, pause, resume, negotiate, or handle ambiguity.

That is why A2A exists.

### 4. UI generation needs a safety model

Generative UI sounds exciting, but raw code generation inside user interfaces is dangerous. A2UI's catalog/rendering approach feels like a better boundary.

The agent can decide the structure, but the application controls the actual components.

### 5. Payment is not just another tool call

An agent making a payment is a high-risk action. AP2's mandate idea is useful because it keeps the human in control through pre-approved rules.
